⓵ O problema que esta tela resolve

1. Target iSCSI cai (rede, controladora, switch)
→
2. Kernel espera replacement_timeout (default 120 s)
→
3. Processos com IO pendente entram em D-state irrecuperável
→
4. qemu trava, iscsiadm trava, host requer reboot

Solução: reduzir replacement_timeout a 15 s + noop_out agressivo. Detecta path morto em ~10 s e libera IO com -EIO em vez de travar para sempre.

⓻ Checklist de troubleshooting

  1. Sessão presa em D-state após queda do target? Não funciona iscsiadm --logout direto — espera o replacement_timeout expirar. Se já travou, agora só reboot. Prevenir: aplicar perfil PVE antes que aconteça.
  2. Aplicou perfil mas continua travando? As mudanças valem apenas para sessões novas. Faça iscsiadm -m node -T <iqn> -p <portal> --logout e depois login, ou reinicie iscsid em janela de manutenção.
  3. "connect failed" no journal? Rede ou firewall. Teste com nc -vz <ip> 3260 ou telnet <ip> 3260.
  4. "Sense Key : Medium Error"? Problema no storage backend (disco com bad sector, RAID degradado). Verifique no servidor iSCSI.
  5. VMs travando em IO mas target OK? Pode ser fila SCSI saturada (queue_depth) ou ARP cache. Verifique iostat -x no host e iscsiadm -m session -P 3 para o estado interno.
  6. Configure multipath se possível. Com 2 paths para o mesmo LUN (2 NICs, 2 portais), queda de um não interrompe IO. Edite /etc/multipath.conf e use multipath -ll para listar.
  7. Configure werror=stop,rerror=stop nas VMs: em vez de travar com IO error, a VM pausa e dá pra retomar quando o storage volta. Adicione no vm-NNN.conf: scsi0: storage:vm-100-disk-0,werror=stop,rerror=stop.
